# Notifications System Implementation Plan

## Overview

Implement the complete notifications system for Workflow Manager: a persisted `Notification` model in the database, a bell icon with unread count badge and dropdown in the dashboard header, real-time delivery via SSE to the frontend, and email notifications sent to workflow recipients when events trigger. This closes the three remaining gaps against the PROJECT_GUIDE.md requirements.

## Problem Statement

The backend SSE infrastructure exists (`NotificationsService` with RxJS Subject, `NotificationsController` with `@Sse('sse')`) but broadcasts to all clients with no user filtering and no frontend consumer. `ResendEmailService` exists and works for the daily summary, but `EventsService.trigger()` does not send emails to workflow recipients. There is no `Notification` database model, no notification bell/dropdown in the UI, and no mechanism to persist or mark notifications as read.

## Solution Architecture

1. **Persisted Notification model** -- New Prisma model with `userId`, `type`, `title`, `body`, `isRead`, `metadata` JSON, and timestamps. tRPC procedures for listing, marking as read (single + all), and unread count.
2. **User-scoped SSE (module abstraction)** -- Refactor `NotificationsService` into a clean module facade. External-facing API: `send({ userId, type, title, body, metadata })` — callers don't know about SSE or DB internals. Internally, maintains a `Map<userId, Subject[]>` for user-scoped SSE delivery and persists to DB. The module is self-contained: any service in the app injects `NotificationsService` and calls `send()` to deliver notifications. Refactor `NotificationsController` to extract the authenticated user from the httpOnly cookie and subscribe to their specific stream.
3. **Email on event trigger** -- After `EventsService.trigger()` creates an event, read the workflow's `recipients` JSON, filter for `EMAIL` channel entries, and send emails via `ResendEmailService` in a fire-and-forget pattern (errors logged, never block event creation). In-app notifications go through `NotificationsService.send()`.
4. **Frontend SSE abstraction** -- Two-layer hook architecture: (a) Generic `useSSE` hook that manages EventSource lifecycle (connect, reconnect via native auto-reconnect, cleanup, connection state) — reusable for any SSE endpoint. (b) `useNotifications` hook that wraps `useSSE` specifically for `/notifications/sse` and invalidates notification tRPC queries on message. Future features (live dashboard, workflow status) can build their own hooks on `useSSE`.
5. **Bell UI** -- Bell icon in the dashboard header with unread count badge. Dropdown (popover) listing recent notifications with mark-as-read actions.

### 1. Add Notification Prisma Model

**File**: `packages/prisma/schema.prisma`

**Current state**: No Notification model exists.

**Modifications**:

- Add `NotificationType` enum with values `EVENT_TRIGGERED`, `EVENT_RESOLVED`, `EVENT_SNOOZED`, `EVENT_REOPENED`
- Add `Notification` model with fields:
  - `id` (cuid)
  - `type` (NotificationType)
  - `title` (String)
  - `body` (String)

  - `userId` (relation to User)
- Add `notifications` relation to User model
- Add indexes on `userId`, `isRead`, and composite `[userId, isRead]`
- Map table to `notifications`

### 3. Refactor NotificationsService for User-Scoped SSE

**File**: `apps/backend/src/features/notifications/notifications.service.ts`

**Current state**: Single `Subject<MessageEvent>` broadcasting to all connected clients. `notify()` accepts `userId` but does not filter.

**Modifications**:

- Replace single `Subject` with a `Map<string, Subject<MessageEvent>[]>` keyed by userId
- Add `subscribe(userId: string): Observable<MessageEvent>` -- creates a new Subject for the user, adds to map, returns observable. On unsubscribe (teardown), remove Subject from map.
- Update `notify(userId: string, event: string, payload)` -- look up user's Subject list and emit to each. If no subjects, notification is simply not pushed (user is offline).
- Add `send(params: { userId: string, type: NotificationType, title: string, body: string, metadata: { eventId: string, workflowId: string } })` -- the **public-facing facade method**. Persists to DB via Prisma AND pushes via SSE. This is the single method external services call. Inject `PrismaService`.
- Add `findAllForUser(userId: string, pagination: PaginationInput, unreadOnly?: boolean)` -- paginated query with optional unread filter
- Add `getUnreadCount(userId: string)` -- returns count of unread notifications
- Add `markAsRead(id: string, userId: string)` -- sets `isRead = true`, validates ownership
- Add `markAllAsRead(userId: string)` -- batch update all unread for user

### 4. Refactor NotificationsController for Auth

**File**: `apps/backend/src/features/notifications/notifications.controller.ts`

### 8. Create Generic useSSE Hook

**File**: `apps/frontend/src/lib/use-sse.hook.ts` (new)

Generic, reusable hook for SSE connections:

- Accepts `{ url: string, withCredentials?: boolean, onMessage?: (event: MessageEvent) => void, enabled?: boolean }`
- Creates an `EventSource` connection with the provided options
- Relies on native `EventSource` auto-reconnection (browsers reconnect automatically with ~3s default). No custom exponential backoff needed — only log errors in `onerror` handler.
- Tracks connection state via `readyState`
- Cleans up EventSource on unmount or when `enabled` becomes false
- Returns `{ isConnected: boolean, readyState: number }`
- This hook is generic and NOT notification-specific — it lives in `lib/` so any feature can use it for any SSE endpoint

### 9. Create useNotifications Hook

**File**: `apps/frontend/src/features/notifications/use-notifications.hook.ts` (new)

Notification-specific hook that wraps `useSSE`:

- Calls `useSSE({ url: '${BACKEND_URL}/notifications/sse', withCredentials: true, onMessage, enabled })`
- In `onMessage` callback: invalidates tRPC queries `notifications.unreadCount` and `notifications.list` (using `trpc.useUtils()`)
- Returns `{ isConnected: boolean }` from the underlying `useSSE`
- Future notification-specific logic (e.g., toast popups, sound alerts) can be added here without touching the generic SSE layer

## Technical Considerations

- **Pattern Usage**: Feature-folder pattern for notifications (backend + frontend). tRPC router follows thin router pattern. Independent `ResendModule` already exists and is global -- just inject `ResendEmailService` where needed.
- **Database Impact**: New `Notification` model with indexes on `userId` and `[userId, isRead]` for efficient unread count and list queries. Migration via `prisma migrate dev`.
- **API Design**: New `notifications` tRPC router with standard CRUD-like procedures. SSE endpoint remains a raw NestJS controller (not tRPC, since SSE is a streaming HTTP response). No breaking changes to existing procedures.
- **Security**: SSE endpoint must authenticate via httpOnly cookie (same as tRPC). Notification ownership validated on mark-as-read. Users can only see their own notifications.
- **Error Handling**: Email sending is fire-and-forget (`Promise.allSettled`) -- failures logged but never block event creation. SSE disconnection handled gracefully with frontend reconnection logic.
- **SSE Auth Lifecycle**: JWT is validated once on connection. If the token expires mid-stream, the connection stays alive until it drops (network error, server restart, etc.). On reconnect, the native `EventSource` auto-reconnection triggers a new HTTP GET, which re-validates the JWT. If the token has expired and cannot be refreshed, the SSE endpoint returns 401 and the frontend should handle this gracefully (redirect to login or show a reconnection state). This is acceptable and does not require mid-stream re-validation.
- **Notification Cleanup (Future)**: No TTL or archival mechanism is included in this iteration. Over time, users will accumulate thousands of notifications. A future iteration should add a cron job to delete or archive notifications older than N days, or a `DELETE /notifications/old` procedure. Noted as a known gap.
- **Edge Cases**:
  - User has no SSE connection (offline): notification persisted in DB, visible on next page load
  - Multiple browser tabs: each tab opens its own EventSource, all receive the SSE event, React Query deduplicates the refetch
  - Workflow with no recipients: in-app notification still created for workflow owner
  - Resend API key not configured: emails silently skipped (existing graceful degradation in `ResendEmailService`)
  - Large number of notifications: paginated queries prevent loading all at once
  - SSE connection drops: frontend reconnects with exponential backoff
  - Mark-as-read race condition: idempotent (setting `isRead=true` on already-read notification is a no-op)
