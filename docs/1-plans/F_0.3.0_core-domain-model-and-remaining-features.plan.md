# Core Domain Model & Remaining Features Implementation Plan

## Overview

Close the fundamental gap between the PROJECT_GUIDE.md requirements and the current Workflow model by adding trigger configuration (threshold/variance), output message templates with variable substitution, and recipient management. Additionally, complete the three extra features (snooze UI polish, comments UI refinement, daily summary cron) and add missing functionality (manual trigger simulation, history page with workflow filter + resolve action).

## Problem Statement

The current Workflow model only stores `{name, description, isActive}`. The PROJECT_GUIDE.md requires workflows to have a **trigger type** (threshold or variance) with typed configuration, an **output message** template supporting `{{metrica}}`/`{{valor}}` variables, and one or more **recipients** (in-app or email channel). Without these, the app cannot evaluate trigger conditions, render alert messages, or route notifications to recipients. The history page lacks workflow filtering, pagination, and inline resolve. The snooze UI is bare (raw datetime input), and comments exist but are not in a polished dialog. No daily summary cron or email module exists.

## Solution Architecture

1. **Prisma schema expansion** -- Add `triggerType` enum (`THRESHOLD`, `VARIANCE`), plus JSON columns `triggerConfig`, `outputMessage`, and `recipients` to the Workflow model. Use `db push` + re-seed for dev environment (no formal migration baseline needed for dev).
2. **Zod discriminated union** -- Define `triggerConfigSchema` as a discriminated union on `type` field, covering threshold (metric, operator, value) and variance (baseValue, deviationPercentage) configs. Define `recipientSchema` for channel routing.
3. **Backend trigger evaluation** -- New `TriggerEvaluationService` evaluates metric values against workflow config, renders output message templates, and creates events when conditions are met.
4. **Manual trigger simulation** -- New tRPC mutation `workflows.simulate` accepts metric values, evaluates against workflow config, returns result without persisting (dry-run) or optionally triggers.
5. **Workflow form expansion** -- Conditional fields for trigger type, operator/value inputs, message template textarea, recipients management (add/remove rows).
6. **History page enhancement** -- Add pagination, workflow dropdown filter, status filter, and inline resolve button.
7. **Snooze UI completion** -- Replace raw datetime input with a duration picker dialog (15min, 30min, 1h, 2h, custom).
8. **Daily summary cron + Resend module** -- Cron aggregates open events by workflow/status, sends summary email via Resend.

### 4. Create Trigger Evaluation Service

**File**: `apps/backend/src/features/workflows/trigger-evaluation.service.ts` (new)

Create a pure service that:

- Method `evaluate(config: TriggerConfig, metricValue: number)`: returns `{ triggered: boolean, details: string }`
  - For THRESHOLD: evaluates `metricValue [operator] config.value`
  - For VARIANCE: evaluates `Math.abs(metricValue - config.baseValue) / config.baseValue * 100 > config.deviationPercentage`
- Method `renderMessage(template: string, metricName: string, metricValue: number)`: replaces `{{metrica}}` with metricName and `{{valor}}` with metricValue string. Unmatched variables left as-is.

### 5. Add Simulate Mutation to Workflows

**File**: `apps/backend/src/features/workflows/workflows.service.ts`

**Current state**: CRUD + toggleActive + delete methods.

**Modifications**:

- Inject `TriggerEvaluationService` and `EventsService`
- Add `simulate(id: string, metricValue: number, userId: string, dryRun: boolean)` method:
  - Fetch workflow with triggerConfig and outputMessage
  - If no triggerConfig, throw `BAD_REQUEST` ("Workflow has no trigger configuration")
  - Parse triggerConfig with Zod discriminated union for runtime safety
  - Call `TriggerEvaluationService.evaluate()` with config and metricValue
  - Render output message via `TriggerEvaluationService.renderMessage()`
  - If triggered AND `dryRun` is false, call `EventsService.trigger()` to create the event (with deduplication)
  - If `dryRun` is true, skip event creation entirely (evaluate only)
  - Return `{ triggered: boolean, message: string, details: string, dryRun: boolean, alreadyOpen?: boolean }`

## Technical Considerations

- **Pattern Usage**: Feature-folder for daily-summary, modules/ for resend. TriggerEvaluationService is a pure logic service within the workflows feature. Thin router pattern maintained.
- **Database Impact**: Three new columns + one new enum on Workflow. JSON columns for flexibility. `db push` + re-seed for dev (no formal migration baseline). No new indexes needed since queries are by workflow ID (already indexed via primary key).
- **API Design**: New `simulate` mutation on workflows router. Expanded `create`/`update` inputs are backward-compatible (new fields are optional). No breaking changes to existing procedures.
- **Security**: Metric value for simulation validated by Zod (number type). Template rendering uses simple string replacement (no eval/interpolation). Resend API key from env vars.
- **Error Handling**: `simulate` returns structured result (not an error) when condition is not met. Throws `BAD_REQUEST` only for invalid config. `CONFLICT` from duplicate open event is caught and returned as `{ triggered: true, message: ..., alreadyOpen: true }`.
- **Edge Cases**:
  - Workflow with no recipients still triggers events (just no notifications routed)
  - Invalid metric values rejected by Zod at input boundary
  - Template variables not found in message left as-is (no crash)
  - Daily summary with 0 events skips email entirely
  - Variance with baseValue=0: guard against division by zero (treat as infinite deviation, always triggers)
  - Simulate on workflow without triggerConfig returns clear error

## Type Definitions

New shared types (all inferred from Zod schemas in `packages/shared/`):

- `TriggerOperator` - `">" | "<" | ">=" | "<=" | "==" | "!="`
- `ThresholdConfig` - `{ type: "THRESHOLD", metric: string, operator: TriggerOperator, value: number }`
- `VarianceConfig` - `{ type: "VARIANCE", baseValue: number, deviationPercentage: number }`
- `TriggerConfig` - `ThresholdConfig | VarianceConfig` (discriminated union)
- `RecipientChannel` - `"IN_APP" | "EMAIL"`
- `Recipient` - `{ channel: RecipientChannel, destination: string }`
- `SimulateWorkflowInput` - `{ id: string, metricValue: number, dryRun: boolean }`
- `SimulateWorkflowResult` - `{ triggered: boolean, message: string, details: string, dryRun: boolean, alreadyOpen?: boolean }`
- Expanded `CreateWorkflowInput` and `UpdateWorkflowInput` with optional trigger fields

## Backward Compatibility

- All new Workflow fields are optional/nullable. Existing workflows with no trigger config continue to work for CRUD but cannot be simulated.
- Dev approach: `prisma db push` to sync schema + `prisma db seed` to re-seed. No formal migration needed for dev environment.
- Existing tRPC procedures are unchanged. New `simulate` mutation is additive.
