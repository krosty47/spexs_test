# Event Trigger Logic Enhancements (Day 4) Implementation Plan

## Overview

Enhance the event trigger flow with workflow active validation, EventHistory creation on trigger, and notification wiring. Add a cron job to auto-reopen events whose snooze has expired. Improve the frontend EventList with working pagination controls. This builds on top of the Day 1-3 scaffold (v0.1.0) and represents the first feature increment.

## Problem Statement

The current `EventsService.trigger()` method creates events without verifying that the parent workflow is active, does not create an EventHistory entry for the TRIGGERED action, and does not notify via `NotificationsService`. Snoozed events remain in SNOOZED status indefinitely even after their `snooze.until` date passes. The frontend EventList is hardcoded to page 1 with no pagination controls.

## Solution Architecture

1. **Workflow active validation** -- Before creating an event in `trigger()`, look up the workflow and verify `isActive === true`. Throw `BAD_REQUEST` if inactive.
2. **TRIGGERED history entry** -- Wrap the event creation in a `$transaction` that also creates an EventHistory with action `'TRIGGERED'`. Since trigger is not user-initiated (it is an external/system call), store a system userId or accept an optional `triggeredBy` field.
3. **Notification on trigger** -- Inject `NotificationsService` into `EventsService` and call `notify()` after successful event creation.
4. **Snooze expiration cron** -- A new `SnoozeExpirationService` runs on `@Cron('*/5 * * * *')` (every 5 minutes), queries for SNOOZED events with `snooze.until <= now()`, and re-opens them (status -> OPEN, creates REOPENED history, deletes snooze record).
5. **Frontend pagination** -- Add page state and prev/next buttons to EventList, wired to the existing paginated tRPC query.

## Implementation Details

### 1. Enhance EventsService.trigger()

**File**: `apps/backend/src/features/events/events.service.ts`

**Current state**: `trigger()` checks for duplicate OPEN events, then creates the event. Does not validate workflow active status, does not create EventHistory, does not notify.

**Modifications**:

- Add `NotificationsService` to constructor injection (around line 1-5)
- In `trigger()` (around line 77-99):
  - Before the deduplication check, query the workflow by `input.workflowId` and verify `isActive === true`. If not found, throw `NOT_FOUND`. If not active, throw `BAD_REQUEST` with message "Workflow is not active".
  - If `snooze.until` is in the past at trigger time for an existing SNOOZED event on the same workflow, treat it as no duplicate (the expired snooze should not block a new trigger).
  - Wrap event creation in `$transaction` to also create an EventHistory entry with action `'TRIGGERED'`.
  - After the transaction, call `this.notificationsService.notify()` with event type `'event.triggered'` and the created event data.
  - Accept an optional `triggeredBy` parameter (userId string) for the history entry. If not provided, use a system constant `'SYSTEM'`.

### 4. Snooze Expiration Cron Service

**File**: `apps/backend/src/features/events/snooze-expiration.service.ts` (new)

Create a new service with `@nestjs/schedule` `@Cron()` decorator:

- Method `handleExpiredSnoozes()` runs every 5 minutes
- Queries for events with status `SNOOZED` and `snooze.until <= new Date()`
- For each expired snooze, in a transaction:
  - Update event status to `OPEN`
  - Create EventHistory entry with action `'REOPENED'` and userId `'SYSTEM'`
  - Delete the Snooze record
- Call `NotificationsService.notify()` for each reopened event with type `'event.reopened'`
- Uses `findMany` with a batch approach (not one-at-a-time) for the query, then processes each in a transaction

## Technical Considerations

- **Pattern Usage**: Follows existing feature-folder pattern. `SnoozeExpirationService` lives inside the events feature since it is tightly coupled to event lifecycle. Thin router pattern maintained -- new logic stays in services.
- **Database Impact**: No new models. One new index on `Snooze.until`. Requires `prisma migrate dev` to generate migration for the index.
- **API Design**: No new tRPC procedures. The `trigger` mutation behavior changes (adds validation + history), which is backward-compatible since it was already creating the event.
- **Security**: Trigger still requires auth (protected procedure). The `triggeredBy` comes from JWT context, not user input.
- **Error Handling**: New error cases use existing tRPC error codes (`BAD_REQUEST` for inactive workflow, `NOT_FOUND` for missing workflow). Consistent with existing patterns.
- **Edge Cases**:
  - **Snooze.until in the past at trigger time**: The trigger should not treat an expired-snoozed event as a "duplicate OPEN event". The deduplication check should only match `status: 'OPEN'` (current behavior is correct). The snooze expiration cron will separately handle reopening these.
  - **Workflow deactivated while events are OPEN**: Existing OPEN events are not affected by workflow deactivation. Only new triggers are blocked. This is the expected behavior -- deactivating a workflow prevents future alerts but does not auto-resolve existing ones.
  - **Cron race condition**: If two cron executions overlap (unlikely with 5-min interval), the transaction ensures each event is only processed once. The `findMany` query might return already-processing events, but the transaction update will be idempotent (updating OPEN to OPEN is a no-op in the worst case, but more precisely, the second transaction will find the event already in OPEN status and the snooze already deleted).
- **Monorepo**: Adding `@@index` to Prisma schema requires regenerating the client (`db:generate`) and running migrations. This affects both apps but is non-breaking.
