# Full-Stack Audit & Hardening Implementation Plan

## Overview

Comprehensive security audit and hardening of the Workflow Manager monorepo across 8 dimensions: removing unused Redis infrastructure, adding security middleware (body limits, Helmet CSP, strict CORS, rate limiting), auditing end-to-end type safety, verifying separation of responsibilities, strengthening error handling, auditing tRPC layer structure, adding comment input sanitization, and a final verification pass.

## Problem Statement

The codebase has matured through v0.5.0 (type safety) and v0.6.0 (notifications/email), but several security and architectural gaps remain:

1. **Redis in infrastructure but unused** -- docker-compose defines a Redis service and env vars reference `REDIS_URL`, but zero application code imports or connects to Redis. This creates confusion and wasted resources.
2. **Security gaps in main.ts** -- No body size limit (DoS vector), Helmet registered without CSP configuration, CORS uses a single string origin, no rate limiting on auth endpoints, and no tRPC error formatter to strip stack traces in production.
3. **Type safety mostly complete** -- v0.5.0 added output schemas and `inferRouterOutputs`. Audit needed to confirm all procedures have input+output schemas, all shared schemas are used, and no manual types remain.
4. **Separation of responsibilities** -- `UsersRouter` has Prisma queries directly in the router (violates thin-router principle). Need to verify all routers delegate to services.
5. **Error handling gaps** -- Global exception filter exists but doesn't handle `TRPCError`. No tRPC error formatter strips stack traces in production. Frontend has toast/error boundary but audit needed for completeness.
6. **tRPC layer structure** -- Context has `[key: string]: unknown` index signature (already noted in v0.5.0 plan as done, but still present). Auth middleware works but could use typing improvements.
7. **Comment XSS** -- Comments are rendered as `{c.content}` in JSX (React auto-escapes, so this is safe for rendering). However, the `addCommentSchema` has no `.trim()` or length normalization on the content field. Backend stores raw user input -- need sanitization at the input boundary for defense-in-depth.
8. **No final verification checklist** exists.

## Solution Architecture

Execute each of the 8 tasks sequentially. Some items discovered during audit may already be correctly implemented (per v0.5.0 work). The plan documents what needs to change and what can be confirmed as already done.

### Approach

- **Task 1 (Redis removal)**: Remove Redis from docker-compose, env files, and env validation schema.
- **Task 2 (Security hardening)**: Add `bodyLimit` to Fastify adapter, configure Helmet with CSP directives, make CORS parse a comma-separated origin list, add `@nestjs/throttler` for auth rate limiting, add tRPC error formatter.
- **Task 3 (Type safety audit)**: Verify all 23 procedures have Zod input/output schemas. Confirm `inferRouterOutputs` usage on frontend. Check shared schemas are complete.
- **Task 4 (Separation of responsibilities)**: Extract `UsersRouter` Prisma query into a `UsersService`. Verify all other routers are thin.
- **Task 5 (Error handling audit)**: Verify TRPCError codes in all services. Review global exception filter. Confirm frontend mutation error handling and error boundary.
- **Task 6 (tRPC layer structure)**: Remove `[key: string]: unknown` from `AppContextType`. Verify `protectedProcedure` pattern (via `AuthMiddleware`). Verify router merging and frontend client setup.
- **Task 7 (Comment sanitization)**: Add `.trim()` and `.transform()` to `addCommentSchema` content field for basic sanitization. Optionally add server-side HTML stripping.
- **Task 8 (Final verification)**: Run builds, tests, and create a checklist confirming all tasks.

### 2. Security Hardening in main.ts (Task 2)

**File**: `apps/backend/src/main.ts`

**Current state**: Helmet registered with no options. CORS uses single `process.env.CORS_ORIGIN` string. No body limit. No rate limiting mention.

**Modifications**:
- Add `bodyLimit: 1_048_576` (1MB) to FastifyAdapter constructor options
- Configure Helmet with CSP: `contentSecurityPolicy: { directives: { defaultSrc: ["'self'"], scriptSrc: ["'self'"], styleSrc: ["'self'", "'unsafe-inline'"], imgSrc: ["'self'", "data:"] } }`. In development, set `contentSecurityPolicy: false` to avoid blocking dev tools.
- Change CORS to parse comma-separated origins: `origin: (process.env.CORS_ORIGIN || 'http://localhost:3000').split(',').map(o => o.trim())`
- Keep `credentials: true`

**File**: `apps/backend/package.json`

**Modifications**:
- Add `@nestjs/throttler` dependency

**File**: `apps/backend/src/app.module.ts`

**Current state**: No ThrottlerModule import.

**Modifications**:
- Import and register `ThrottlerModule.forRoot([{ ttl: 60000, limit: 10 }])` with default global rate limiting
- The default is permissive (10 requests per 60s); auth endpoints will have stricter limits

**File**: `apps/backend/src/features/auth/auth.router.ts`

**Current state**: No rate limiting on login/register.

**Modifications**:
- Note: `@nestjs/throttler` works with NestJS guards, and nestjs-trpc routes go through tRPC middleware, not standard NestJS guards. The ThrottlerModule guard approach won't directly apply to tRPC procedures. Instead, implement a simple in-memory rate limiter as a tRPC middleware for auth procedures.
- Create `apps/backend/src/trpc/rate-limit.middleware.ts` -- a tRPC middleware that tracks requests per IP per window using a Map. Apply to auth router's login/register/refresh mutations (5 requests per 60 seconds for login/register).

**File**: `apps/backend/src/trpc/rate-limit.middleware.ts` (new)

A NestJS injectable that implements `TRPCMiddleware` from nestjs-trpc. Tracks request counts per IP using an in-memory Map with TTL-based cleanup. Throws `TRPCError` with code `TOO_MANY_REQUESTS` when limit exceeded.

**File**: `apps/backend/src/trpc/trpc.module.ts`

**Modifications**:
- Register `RateLimitMiddleware` as a provider and export

**File**: `apps/backend/src/features/auth/auth.router.ts`

**Modifications**:
- Apply `@UseMiddlewares(RateLimitMiddleware)` to `login`, `register`, and `refresh` mutations

**File**: `apps/backend/src/trpc/trpc.module.ts` or `apps/backend/src/trpc/context.ts`

For tRPC error formatter -- nestjs-trpc's `TRPCModule.forRoot()` supports an `errorFormatter` option (or it may need to be configured via the module). If nestjs-trpc doesn't support `errorFormatter` in its config, the error formatting can be done in the `GlobalExceptionFilter` or via a custom tRPC middleware.

**Modifications**:
- In `TRPCModule.forRoot()`, add an error formatter that strips stack traces when `NODE_ENV === 'production'`
- If nestjs-trpc doesn't support the `errorFormatter` option, create a tRPC error-logging middleware that catches errors and strips stack traces before re-throwing

### 7. Comment Input Sanitization (Task 7)

**Analysis**: React's JSX rendering auto-escapes content (`{c.content}`), so XSS via comment rendering is not possible in the current frontend. However, defense-in-depth requires sanitizing at the input boundary.

**File**: `packages/shared/src/schemas/event.schema.ts`

**Current state**: `addCommentSchema` has `content: z.string().min(1).max(2000)`.

**Modifications**:
- Add `.trim()` to normalize whitespace
- Add `.transform()` to strip HTML tags as basic sanitization: `.transform((val) => val.replace(/<[^>]*>/g, ''))`
- Update `.min(1)` to be checked after trim (move min after trim by using `.pipe()` or restructuring)
- The schema becomes: `content: z.string().trim().min(1, 'Comment cannot be empty').max(2000).transform((val) => val.replace(/<[^>]*>/g, ''))`

**File**: `apps/frontend/src/features/events/event-detail.component.tsx`

**Current state**: Comment content is rendered via `{c.content}` -- already safe with React's auto-escaping.

**No frontend changes needed for XSS prevention.**

## Technical Considerations

- **Pattern Usage**: All changes follow existing patterns (feature-folder structure, thin routers, Zod schemas in shared package, tRPC middleware via nestjs-trpc)
- **Database Impact**: None. No schema changes or migrations needed.
- **API Design**: No breaking changes. Output schemas already exist. Input schema changes (trim/sanitize) are non-breaking.
- **Security**: Body limit prevents large-payload DoS. Helmet CSP prevents XSS via injected scripts. CORS whitelist prevents cross-origin abuse. Rate limiting prevents brute-force auth attacks. Error formatter prevents stack trace leakage. Comment sanitization provides defense-in-depth.
- **Error Handling**: Error formatter ensures production users never see stack traces. GlobalExceptionFilter improvements handle edge cases.
- **Edge Cases**:
  - Rate limiter in-memory state is lost on restart (acceptable for single-instance deployment)
  - Helmet CSP may break if frontend loads external resources (configure directives carefully)
  - CORS comma-separated parsing must handle whitespace
  - Comment sanitization regex is basic -- doesn't handle encoded HTML entities, but React's auto-escaping covers rendering
