# Scaffold Monorepo (Days 1-3) Implementation Plan

## Overview

Scaffold the complete Workflow Manager monorepo covering the first three development days: Day 1 sets up the Turborepo + pnpm monorepo, Prisma schema with all models, Docker Compose (PostgreSQL + Redis), and seed data. Day 2 builds the NestJS backend with Fastify, tRPC v11 routers (workflows, events), and JWT auth with httpOnly cookies. Day 3 creates the Next.js 15 frontend with tRPC client, shadcn/ui, workflow pages, and Zod-validated forms.

## Problem Statement

The project currently has only documentation files and no application code. We need to scaffold a working end-to-end application following the architecture defined in ARCHI.md and the conventions in BEST-PRACTICES.md.

## Solution Architecture

A Turborepo monorepo with two apps (`backend`, `frontend`) and two shared packages (`prisma`, `shared`). The backend uses NestJS + Fastify with tRPC v11 for type-safe API communication. The frontend uses Next.js 15 App Router with tRPC client and shadcn/ui. PostgreSQL and Redis run in Docker containers for local development. Authentication uses JWT stored in httpOnly cookies via Passport. Zod is the single validation library across the stack.

### 3. Prisma Package

**File**: `packages/prisma/package.json`

Package config with name `@workflow-manager/prisma`, scripts for `db:generate`, `db:push`, `db:seed`, `db:migrate`. Dependencies: `prisma`, `@prisma/client`, `tsx` (for seed script).

**File**: `packages/prisma/tsconfig.json`

TypeScript config extending root `tsconfig.base.json`.

**File**: `packages/prisma/schema.prisma`

Complete schema with all six models following ARCHI.md conventions:

- **User**: id (cuid), email (unique), name, passwordHash, role (enum: ADMIN/USER), createdAt, updatedAt. Mapped to `users`.
- **Workflow**: id (cuid), name, description (optional), isActive (default false), createdAt, updatedAt, userId (FK). Relations: events[], user. Mapped to `workflows`.
- **Event**: id (cuid), title, payload (Json), status (enum: OPEN/RESOLVED/SNOOZED), createdAt, updatedAt, workflowId (FK), resolvedAt (optional), resolvedById (optional). Relations: workflow, comments[], snooze (optional), history[], resolvedBy (optional User). Mapped to `events`.
- **EventHistory**: id (cuid), action (string), createdAt, eventId (FK), userId (FK). Relations: event, user. Mapped to `event_history`.
- **Comment**: id (cuid), content (string), createdAt, updatedAt, eventId (FK), userId (FK). Relations: event, user. Mapped to `comments`.
- **Snooze**: id (cuid), until (DateTime), reason (optional string), createdAt, eventId (unique FK), userId (FK). Relations: event, user. Mapped to `snoozes`.

Enums: `Role` (ADMIN, USER), `EventStatus` (OPEN, RESOLVED, SNOOZED).

All fields use `@map("snake_case")` for database columns, all models use `@@map("table_name")`.
Database indexes on: `Event.status`, `Event.workflowId`, `Event.createdAt`, `Workflow.isActive`, `Workflow.userId`, `EventHistory.eventId`.

**File**: `packages/prisma/seed.ts`

Seed script using `upsert` for idempotency:

- 2 users (admin + regular user) with bcrypt-hashed passwords
- 3 sample workflows (active and inactive)
- 5 sample events across workflows (mix of OPEN, RESOLVED, SNOOZED)
- Sample event history entries
- Sample comments and snooze records

## Technical Considerations

- **Pattern Usage**: Follow feature-folder pattern from ARCHI.md. Thin routers delegating to services. Independent external modules in `src/modules/`.
- **Database Impact**: Full schema creation with 6 models, enums, indexes on frequently queried fields. Use `prisma migrate dev` for initial migration.
- **API Design**: tRPC v11 with nestjs-trpc decorators (`@Router()`, `@Query()`, `@Mutation()`). Root router merges feature routers. AppRouter type exported for frontend consumption.
- **Security**: httpOnly cookies for JWT (SameSite=Strict, Secure in prod). Zod validation on every tRPC input. bcrypt for password hashing (cost >= 12). Global JWT guard with `@Public()` bypass.
- **Error Handling**: tRPC error codes mapped to domain exceptions. Global exception filter for unhandled errors.
- **Edge Cases**: Duplicate email on registration, event deduplication (no duplicate OPEN events per workflow), expired JWT handling with refresh flow, missing required env vars at startup.
- **Monorepo**: Changes to `packages/prisma` or `packages/shared` affect both apps. Turborepo handles build ordering via `dependsOn`.
