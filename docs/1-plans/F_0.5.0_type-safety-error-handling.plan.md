# End-to-End Type Safety & Error Handling Hardening Plan

## Overview

Harden the Workflow Manager monorepo with end-to-end type safety improvements and error handling refinements. This covers 7 items: defining output Zod schemas on all tRPC procedures, replacing manual frontend types with inferred tRPC types, typing Prisma JSON fields, adding a React Error Boundary, converting ResendEmailService errors to TRPCError, tightening the tRPC context type, and adding a toast notification system for frontend errors.

## Problem Statement

The current codebase has several type safety gaps despite using tRPC for end-to-end types:

1. **No output schemas** on tRPC procedures -- tRPC infers return types from service implementations but does not validate them at runtime. If a service accidentally returns extra or differently-shaped data, it flows to the frontend unchecked.
2. **Manual frontend types** -- `apps/frontend/src/features/workflows/workflow.types.ts` defines a hand-written `Workflow` interface that duplicates (and can drift from) the actual backend return shape. tRPC's `inferRouterOutputs` is not used.
3. **Untyped JSON fields** -- Prisma's `Json` type for `triggerConfig`, `recipients`, `payload`, and `metadata` forces `as unknown as` casts and provides no compile-time or runtime guarantees on these fields' shapes.
4. **No React Error Boundary** -- Unhandled rendering errors crash the entire app. The architecture doc (section 19) states error boundaries should wrap feature sections.
5. **ResendEmailService throws bare `Error`** -- Lines 59 and 82 of `resend-email.service.ts` throw `new Error(...)` instead of `TRPCError`, violating the project's error handling convention (BEST-PRACTICES section 1).
6. **Loose tRPC context typing** -- `AppContextType` has `[key: string]: unknown` index signature, and `getCookie` uses an `as unknown as` cast for the request cookies.
7. **No toast system** -- Frontend errors from tRPC mutations are silently swallowed or logged to console. No user-visible feedback mechanism exists.

## Solution Architecture

### Approach

1. Define output Zod schemas in `packages/shared/src/schemas/` (one per domain: workflow, event, auth, notification). These schemas describe the exact shape returned by each tRPC procedure.
2. Add the `output` field to every `@Query()` and `@Mutation()` decorator in the backend routers. nestjs-trpc v2.9.1 supports `@Query({ input: ..., output: ... })`.
3. On the frontend, create a `trpc-types.ts` utility that exports inferred types via `inferRouterOutputs<AppRouter>`. Delete `workflow.types.ts` and update all consumers.
4. For Prisma JSON fields, add branded/refined Zod schemas and use `.parse()` at the service layer boundary (where data exits Prisma) rather than `as unknown as` casts. The output schemas handle the serialization boundary.
5. Add an `ErrorBoundary` component wrapping feature sections in the dashboard layout.
6. Add a `Toaster` component (shadcn/ui toast or sonner) for mutation error feedback.
7. Fix `ResendEmailService` to throw `TRPCError` with `INTERNAL_SERVER_ERROR`.
8. Remove the index signature from `AppContextType` and properly type the Fastify request cookies.

### Date Serialization Strategy

Prisma returns `Date` objects; tRPC serializes them to ISO strings over the wire. Output schemas must use `z.coerce.date()` or `z.string().datetime()` depending on whether the schema is used for runtime validation (backend) vs. type inference (frontend). The approach: use `z.coerce.date()` in output schemas so they work on both sides -- Prisma Date objects pass, and stringified dates from the client also parse.

### 4. Type Prisma JSON Fields

**File**: `apps/backend/src/features/workflows/workflows.service.ts`

**Current state**: `triggerConfig` and `recipients` are typed as `Json` by Prisma. Line 99 uses `input.triggerConfig as Prisma.InputJsonValue` and line 105 uses `input.recipients as unknown as Prisma.InputJsonValue`.

**Modifications**:

- After fetching workflows from Prisma, parse JSON fields with Zod schemas at the service boundary before returning. For `findAll` and `findOne`, parse `triggerConfig` with `triggerConfigSchema.nullable().parse()` and `recipients` with `z.array(recipientSchema).parse()`.
- This eliminates the need for `as unknown as` casts on output. For input, the casts to `Prisma.InputJsonValue` are acceptable since they go INTO Prisma (Zod already validated the input shape).
- Remove the `as unknown as` cast on line 105 by using a direct cast to `Prisma.InputJsonValue` (the Zod-validated input is already the correct shape).

**File**: `apps/backend/src/features/events/events.service.ts`

**Current state**: `payload` is typed as `Json` by Prisma. Line 132 uses `input.payload as Prisma.InputJsonValue`.

**Modifications**:

- The `payload` field is intentionally schemaless (`z.record(z.unknown())`), so no runtime parsing is needed on output. The output schema will use `z.record(z.unknown())` which accepts any JSON object.
- The input cast on line 132 is acceptable (Zod validated the shape).

### 6. ResendEmailService TRPCError

**File**: `apps/backend/src/features/resend/services/resend-email.service.ts`

**Current state**: Line 59 throws `new Error('No "from" address provided...')` and line 82 throws `new Error('Failed to send email: ...')`.

**Modifications**:

- Import `TRPCError` from `@trpc/server`
- Line 59: Change to `throw new TRPCError({ code: 'INTERNAL_SERVER_ERROR', message: 'No "from" address provided...' })`
- Line 82: Change to `throw new TRPCError({ code: 'INTERNAL_SERVER_ERROR', message: 'Failed to send email: ...' })`

Note: While `ResendEmailService` is an independent module (no tRPC dependency per BEST-PRACTICES section 10), the convention states "Never use bare `throw new Error()` -- always include context." Since this service is only consumed within the NestJS app which runs in a tRPC context, using `TRPCError` is acceptable. Alternatively, a custom `EmailSendError` class could be used and mapped at the caller. The simpler approach (TRPCError directly) is chosen since there are no plans to extract this module.

## Technical Considerations

- **Pattern Usage**: Output schemas follow the existing shared schema pattern in `packages/shared/src/schemas/`. Router changes are minimal (adding `output` field to decorators). Frontend type migration uses tRPC's built-in `inferRouterOutputs`.
- **Database Impact**: None. No schema changes or migrations needed.
- **API Design**: Output schemas must match current return shapes exactly. If a schema is too strict, the procedure will fail at runtime. Run existing tests after adding output schemas to catch mismatches.
- **Security**: Output schemas provide an additional safety net -- they strip any accidental data leaks (e.g., password hashes) before data reaches the client.
- **Error Handling**: Converting `ResendEmailService` to `TRPCError` ensures consistent error types. Error Boundary catches rendering errors. Toast system surfaces mutation errors to users.
- **Edge Cases**:
  - **Date serialization**: Prisma `Date` objects must pass through output schemas. Using `z.coerce.date()` handles both `Date` objects (backend) and ISO strings (frontend).
  - **Null JSON fields**: `triggerConfig` and `recipients` can be null in the database. Output schemas must use `.nullable()` for these fields.
  - **Empty recipients**: `recipients` defaults to `[]` in Prisma but the column is `Json?`. Schema must handle both `null` and empty array.
  - **`markAsRead` returns null**: When using `findUnique` after `updateMany`, it can theoretically return null. Output schema uses `.nullable()`.
  - **Breaking existing tests**: Adding output schemas may cause test failures if mocked return values don't exactly match the schema. Mock data must be updated.
- **Backward Compatibility**: Output schemas are purely additive -- they validate data that is already being returned. If schemas match the current shapes (which they must), no API behavior changes. The frontend type migration is also non-breaking since inferred types will match the new output schemas.

## Type Definitions

New types defined via Zod schemas in `packages/shared/`:

- `WorkflowOutput` -- Full workflow shape for single-item responses
- `WorkflowListOutput` -- Paginated workflow list response
- `WorkflowDetailOutput` -- Workflow with events and count
- `SimulateWorkflowOutput` -- Simulation result (replaces `SimulateWorkflowResult` interface)
- `EventOutput` -- Single event shape
- `EventWithWorkflow` -- Event with workflow name and comment count
- `EventListOutput` -- Paginated event list response
- `EventDetailOutput` -- Event with comments, snooze, history
- `CommentOutput` -- Comment with user info
- `EventHistoryOutput` -- History entry with user info
- `SnoozeOutput` -- Snooze details
- `AuthOutput` -- Login/register response (tokens + user)
- `TrpcUserOutput` -- User from context (id, email, role)
- `NotificationListOutput` -- Paginated notification list
- `MarkAllAsReadOutput` -- Batch update count

Frontend inferred types via `inferRouterOutputs<AppRouter>`:

- `RouterOutputs` -- Full map of all router outputs
- Convenience aliases for frequently used types (Workflow, Event, etc.)

## Performance & Cost Impact

- Output schema validation adds minimal runtime overhead (Zod parse on response). For list endpoints with many items, this is negligible compared to DB query time.
- No database changes, so no query performance impact.
- Toast library (sonner) is lightweight (~3KB gzipped).
